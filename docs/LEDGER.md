# The keel ledger — format specification

`.keel/` at the project root is keel's persistent, human-readable engineering ledger. Plain files, diffable, committable, portable to any harness. This document is the canonical format; the skills read and write these files, and the hooks parse them with `grep`/`sed` — so the machine-checked shapes below (requirement lines, config keys, record headings) must be kept exact.

```
.keel/
  config            # gate toggles — simple KEY=VALUE, bash-parseable
  spec.md           # living Working Spec: goal, constraints, requirement checklist
  plan.md           # current phase plan (optional; written by implementation-planner)
  decisions.md      # append-only decision log (D1, D2, ...)
  verification.md   # append-only verification records (V1, V2, ...)
  handoffs/         # dated handoff notes + mechanical session records
  deadends.md       # append-only: approaches tried and abandoned, and why
  digest.md         # compact regenerated summary — what SessionStart injects
  .gitignore        # ignores state/
  state/            # machine state, never committed
    last-verified   # content fingerprint at the last verification record
```

## config

One `KEY=VALUE` per line, `#` comments allowed, no spaces around `=` (hooks match with `grep '^KEY=1'`):

```
# keel gate toggles — 1 to enable, 0 to disable
GATE_FIRST_EDIT=0
GATE_STOP_VERIFY=0
```

- `GATE_FIRST_EDIT=1` — block the session's first file edit until `spec.md` has at least one requirement (blocks once per session; recorded-skip escape hatch via `decisions.md`).
- `GATE_STOP_VERIFY=1` — block a stop when the session edited files and the working tree no longer matches the last verification record.

## spec.md — the Working Spec

```markdown
# Working Spec
Updated: YYYY-MM-DD

## Goal
<the real outcome, 1–2 sentences>

## Constraints
<hard rules the implementation must respect — re-injected after compaction>

## Requirements
- [ ] R1: Settlement amounts round half-up to 2 decimal places
- [x] R2: Members can be removed only at zero balance  (verified: V7)
- [~] R3: Export to CSV  (dropped 2026-07-15, see D4)

## Assumptions
## Out of scope
## Risks
```

Requirement lines are machine-checked: `- [ ] R<n>: <text>`. States: `[ ]` open · `[x]` met **and verified** (must cite a `V` record) · `[~]` dropped (must cite a `D` record). IDs are stable and never reused — requirements are explicitly closed or dropped, never silently deleted. The `## Goal`, `## Constraints`, and `## Requirements` headings are parsed by the post-compaction re-anchor hook; keep them spelled exactly.

## plan.md

The implementation-planner's phase plan. Free-form phases, but include a current-phase marker line the digest can carry:

```markdown
# Implementation Plan
Updated: YYYY-MM-DD
Current phase: 2 — Minimal implementation
```

## decisions.md — append-only

```markdown
## D4 — 2026-07-15 — Dropped CSV export
Chose to drop R3 for v1 because the reporting service already exports; revisit if users ask.

## D5 — 2026-07-15 — Second-opinion verdict: Revise (settlement rounding)
Blocking: rounding applied per-expense instead of per-settlement (R1). Full review in the session transcript.
```

Entries: `## D<n> — YYYY-MM-DD — <title>`, then a short body with the *why*. Written by second-opinion-review (verdicts), gate skips, and any skill recording a non-obvious choice.

## verification.md — append-only

```markdown
## V7 — 2026-07-15 14:32 UTC — commit a1b2c3d
- Ran: `npm test` (142 passed), `npx tsc --noEmit` (clean), `npm run lint` (clean)
- Verified: R1, R2 via test suite; R2 additionally in browser
- Assumed, not verified: mobile layout; concurrent-edit behavior
```

Written only by verification-gate, which then updates the requirement checkboxes it evidenced and stamps `state/last-verified`.

## handoffs/

- `YYYY-MM-DD-<slug>.md` — rich model-authored notes from agent-handoff-summary.
- `sessions.md` — append-only mechanical records written by the SessionEnd hook (timestamp, git summary, files edited). Deterministic breadcrumbs, not a substitute for the rich notes.

## deadends.md — append-only

```markdown
## 2026-07-15 — PgBouncer transaction mode
Broke prepared statements; reverted. Don't retry without statement-cache changes.
```

One entry per abandoned approach, with enough *why* that no future session retries it blind.

## digest.md — regenerated, never hand-curated history

The compact summary the SessionStart hook injects every session. Regenerated by the skills as they write, and rebuilt from scratch by context-recovery. **Adaptive size, hard cap 600 tokens (~450 words)** — generate the smallest digest the ledger's content needs, dropping from the tail when the cap binds:

```markdown
Goal: <1–2 lines>
Phase: <current phase from plan.md, if any>
Open requirements: R4, R6, R7 (of 7; 4 verified, 1 dropped)
Last verification: V7 (2026-07-15, commit a1b2c3d) — all checks green
Last handoff: 2026-07-15-settlement-rounding.md — <1-line summary>
Recent decisions: D4 <1 line>; D5 <1 line>
Dead ends: don't retry PgBouncer transaction mode
```

Priority order when trimming: goal → open requirements → last verification → phase → last handoff → decisions → dead ends.

## state/last-verified

Machine-written by `stop-gate.sh --record` (never by hand). Line 1: a content fingerprint — hash of the sorted list of all tracked + untracked non-ignored files (excluding `.keel/`) plus their per-file content hashes. Line 2: `recorded=<ISO time> commit=<short>`. Content-based, so committing already-verified changes does not invalidate it, and ledger writes never dirty it.
